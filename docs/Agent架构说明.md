# Agent架构说明

## 1. 架构总览

### 1.1 总体定位

本项目的目标不是构建一个完全自治、相互博弈的多智能体平台，而是围绕“多教材解析、知识图谱生成、整合摘要、RAG问答、教师反馈迭代”这一条教学工作流，构建一个**可运行、可解释、可展示**的学科知识整合系统。

因此，当前实现采用的是：

- **前端单页应用 + 后端分层服务**
- **逻辑 Agent 映射到服务模块**
- **关键长耗时任务采用异步后台执行**

换句话说，当前系统中的“Agent”是**职责清晰的逻辑角色**，并不等价于多个独立进程或多个自治大模型实例。

### 1.2 系统分层

当前代码架构可分为四层：

1. **API 层**
   - 对外提供教材、图谱、分析、问答、报告等 REST 接口
   - 对应目录：`src/backend/server/api/routes`

2. **Service 层**
   - 承担各类 Agent 的核心业务逻辑
   - 对应目录：`src/backend/server/services`

3. **Repository 层**
   - 负责教材、对话、报告等数据的文件化持久化
   - 对应目录：`src/backend/server/repositories`

4. **Client / Utility 层**
   - Client 层负责外部大模型调用
   - Utility 层负责文件解析、文本切分、关键词提取等基础能力
   - 对应目录：
     - `src/backend/server/clients`
     - `src/backend/server/utils`

### 1.3 Agent 角色划分

当前系统中的核心 Agent 角色如下：

#### 1. Material Parsing Agent

职责：

- 接收教材上传
- 支持 `pdf / docx / txt / md`
- 完成逐页解析、章节识别、小节结构化、文本切块

对应实现：

- `TextbookService`
- `file_loader.py`
- `text_processing.py`

#### 2. Knowledge Graph Agent

职责：

- 对每个章节调用 LLM
- 抽取章节级知识点与知识点关系
- 输出单教材图谱与多教材合并图谱

对应实现：

- `GraphService`
- `prompt.md`
- `OpenAICompatibleClient`

#### 3. Curriculum Comparison Agent

职责：

- 识别跨教材共享主题
- 计算教材之间的重叠、互补与缺失

对应实现：

- `AnalysisService`

#### 4. Integration Agent

职责：

- 按目标压缩比例生成整合摘要
- 保留原文来源信息

对应实现：

- `IntegrationService`

#### 5. Retrieval QA Agent

职责：

- 基于教材切块建立可检索语料
- 执行检索召回
- 输出答案与引用

对应实现：

- `QAService`

#### 6. Dialogue Agent

职责：

- 记录教师反馈
- 基于当前分析与摘要结果返回调整建议

对应实现：

- `DialogueService`
- `ConversationRepository`

#### 7. Reporting Agent

职责：

- 汇总教材、分析、摘要、反馈
- 输出 Markdown 报告

对应实现：

- `ReportService`
- `ReportRepository`

## 2. 设计决策论证

### 2.1 为什么采用“逻辑 Agent + 分层服务”，而不是多进程多模型编排

当前项目面向比赛展示与教学场景，更强调：

- 可运行
- 可解释
- 开发复杂度可控
- 前后端联调简单

如果一开始就拆成多进程、多队列、多模型自治架构，会显著增加：

- 调度复杂度
- 错误排查成本
- 演示失败风险

因此，当前项目选择：

- 在设计上保留 Agent 角色边界
- 在工程实现上使用同一 Flask 服务中的多个 Service 模块承载这些职责

这是一种“先保证业务闭环，再逐步拆分”的决策。

### 2.2 为什么正文解析与知识图谱生成解耦

这是当前系统里最关键的架构决策之一。

早期实现中，正文解析和图谱生成耦合在同一条链路上，导致：

- 大文件解析耗时长
- 章节级 LLM 调用易超时
- 一旦图谱生成失败，整本教材会被误判为解析失败

因此当前实现将两者拆开：

1. **教材解析阶段**
   - 只负责文件落盘、逐页解析、章节和文本块生成
   - 解析完成后教材状态记为 `ready`

2. **图谱生成阶段**
   - 单独通过 `/api/graphs/textbooks/{id}/generate` 触发
   - 独立维护 `graph_status / graph_progress / graph_error_message`

这种拆分使系统具备更好的鲁棒性：

- 正文解析成功不会被图谱超时拖垮
- 可以重复生成图谱
- 更适合比赛现场分步演示

### 2.3 为什么采用文件系统持久化，而不是数据库

当前项目的数据规模和访问模式相对简单：

- 教材元数据
- 教材解析结果
- 教师对话
- 报告内容

这些数据都可以稳定地落在：

- `runtime/data`
- `runtime/uploads`
- `runtime/reports`
- `runtime/logs`

采用文件系统持久化的原因：

1. 初版部署成本低
2. 不需要额外数据库容器
3. 便于评委直接查看中间结果
4. 对本项目体量足够

### 2.4 为什么 RAG 采用 TF-IDF 而不是向量数据库

当前 RAG 方案选择：

- 本地 TF-IDF
- 余弦相似度召回

原因不是因为这是理论上的最优解，而是因为它在当前阶段更适合：

1. 本地可运行
2. 依赖少
3. 易解释
4. 对中文教材长文本已具备基本可用性

同时，系统保留了未来升级空间：

- 可以切换为向量检索
- 可以加入 rerank
- 可以接知识图谱增强检索

### 2.5 为什么知识图谱采用“单章节调用 LLM”

当前 `prompt.md` 明确约束：

- 每次只处理一个章节
- 严格输出 JSON
- 控制知识点数量

这样设计的原因：

1. 避免整本教材上下文过长
2. 保持章节语义边界清晰
3. 有利于稳定抽取知识点和 prerequisite 关系
4. 便于失败重试和图谱进度跟踪

## 3. 数据流

### 3.1 教材解析主流程

1. 用户在前端上传教材文件
2. `TextbookService` 保存原始文件到 `runtime/uploads`
3. Repository 写入教材占位记录，状态为 `parsing`
4. 后台线程启动正文解析任务
5. `parse_document_file()` 按文件类型执行解析：
   - PDF：逐页提取、去页眉页脚、章节识别
   - DOCX / TXT / MD：按段或按行解析
6. 解析结果写入：
   - `chapters`
   - `sections`
   - `chunks`
   - `keywords`
7. 教材状态更新为 `ready`

### 3.2 知识图谱生成流程

1. 用户或测试脚本调用图谱生成接口
2. `TextbookService.enqueue_graph_generation()` 提交后台任务
3. `GraphService.build_textbook_graph()` 遍历章节
4. 每个章节调用 LLM，输出：
   - `knowledge_points`
   - `relations`
5. 系统完成：
   - 节点规范化
   - 重复知识点合并
   - 关系去重
6. 图谱结果写回教材详情：
   - `nodes`
   - `edges`
   - `relation_types`
   - `chapter_graphs`
7. `graph_status` 更新为 `ready`

### 3.3 多教材分析流程

1. 用户选择多本已解析教材
2. `AnalysisService` 汇总关键词
3. 计算：
   - shared keywords
   - pairwise similarity
   - unique topics
   - missing topics
4. 若每本教材图谱都已生成，则 `GraphService` 会合并知识点图谱
5. 否则回退为关键词级组合图谱

### 3.4 整合摘要流程

1. `IntegrationService` 从各教材小节提取句子
2. 计算句子词频得分
3. 按预算比例选句
4. 最终按原教材 / 小节 / 句序回排
5. 输出带来源引用的摘要结果

### 3.5 RAG 问答流程

1. `QAService` 读取教材 `chunks`
2. 用 TF-IDF 建立检索矩阵
3. 对用户问题做相似度检索
4. 返回 top-k 原文片段
5. 若启用 LLM，则基于片段生成带引用答案
6. 若未启用，则直接做抽取式回答

### 3.6 教师反馈与报告流程

1. 教师反馈进入 `DialogueService`
2. 反馈记录写入对话存储
3. `ReportService` 汇总：
   - 教材概况
   - 跨教材分析
   - 整合摘要
   - 教师反馈
4. 最终输出 Markdown 报告

## 4. 取舍与权衡

### 4.1 架构上的取舍

#### 选择

- 单 Flask 服务承载所有 Agent 逻辑

#### 收益

- 实现成本低
- 演示稳定
- 便于联调

#### 代价

- 扩展性不如真正的分布式多 Agent 系统
- 后续并发能力受限

### 4.2 持久化上的取舍

#### 选择

- 文件系统 JSON 持久化

#### 收益

- 轻量
- 易查看
- 无数据库依赖

#### 代价

- 不适合高并发写入
- 不适合复杂查询

### 4.3 图谱生成上的取舍

#### 选择

- 单章节调用 LLM
- 图谱生成与正文解析分开

#### 收益

- 失败边界清晰
- 进度可追踪
- 便于重试

#### 代价

- 整体耗时增加
- 章节间全局关系较弱

### 4.4 RAG 实现上的取舍

#### 选择

- 本地 TF-IDF 检索

#### 收益

- 无外部向量服务依赖
- 评委可离线运行
- 检索逻辑透明

#### 代价

- 语义召回能力弱于向量检索
- 对同义表达不够鲁棒

### 4.5 可视化上的取舍

#### 选择

- 前端采用 Cytoscape.js 交互图谱

#### 收益

- 支持缩放、拖拽、节点点击
- 便于展示教材来源和知识点频次

#### 代价

- 前端类型和交互实现复杂度上升
- 超大图谱时前端渲染压力会增加

---

## 5. 结论

当前项目的 Agent 架构不是“多模型自治系统”，而是一个**以业务闭环为导向的逻辑 Agent 架构**。

它的核心特点是：

1. **职责清晰**  
   每个 Agent 对应明确的业务边界：解析、图谱、比较、摘要、问答、反馈、报告。

2. **工程可运行**  
   通过 Flask 分层服务与文件持久化，优先保证完整演示链路。

3. **关键流程解耦**  
   将正文解析与图谱生成拆分，显著提升稳定性。

4. **支持后续演进**  
   后续可以继续向：
   - 任务队列化
   - 向量检索
   - 图谱增强推理
   - 教学路径自动规划
   演进。

因此，这份架构既满足比赛阶段“可展示、可解释、可运行”的目标，也为后续真正的多 Agent 系统化扩展保留了空间。
