# 需求分析

## 1. 项目目标

本项目面向“多教材知识整合”场景，目标是构建一个可运行的学科知识整合智能体，支持：

1. 自动加载多本教材，支持 `PDF / Markdown / TXT / Word.docx`
2. 对教材正文进行结构化解析，形成章节、小节、页码与正文内容
3. 为每本教材分别构建知识图谱，并支持前端可交互可视化
4. 支持跨教材识别知识点的重叠、互补与缺失
5. 在不超过原始体量目标比例的前提下，生成整合摘要
6. 基于教材原文进行 RAG 问答，并返回引用来源
7. 支持教师反馈与分析报告输出

在当前实现中，系统采用“正文解析”和“知识图谱生成”解耦的方式：

- 正文解析先完成文件落盘、逐页处理、章节提取与文本切块
- 知识图谱再单独按章节调用 LLM 生成知识点和关系

这样可以避免大文件解析被 LLM 调用超时拖垮整条流程。

---

## 2. 子问题分解与实际解决方案

### 2.1 知识点粒度定义

#### 需求问题

如果知识点粒度过粗，图谱会退化为“章节级目录图”；  
如果粒度过细，图谱节点会爆炸，关系噪声过多，不利于教学使用。

因此必须先回答：系统中的“知识点”到底定义到什么层级。

#### 当前项目的实际方案

当前实现采用“章节级调用 LLM、章节内抽取核心知识点”的粒度策略。

具体约束如下：

1. 以“单个章节”作为一次 LLM 调用的输入边界，避免上下文过长
2. 每个章节只抽取核心知识点，不追求覆盖所有术语
3. 知识点类型以教学可解释对象为主，包括：
   - 核心概念
   - 定理
   - 方法
   - 现象
   - 关键条件
   - 设计技术
   - 原型滤波器等扩展类别
4. 当前 prompt 约束每章建议抽取 `5~12` 个知识点

#### 当前结构化输出

系统中单个知识点的结构为：

```json
{
  "id": "node_001",
  "name": "动作电位",
  "definition": "细胞受到刺激后，膜电位发生的一次快速而可逆的倒转。",
  "category": "核心概念",
  "chapter": "第二章 细胞的基本功能",
  "page": 35
}
```

#### 这样定义的原因

1. 章节是教材天然的知识组织单元，适合作为抽取边界
2. 章节级粒度足以保留知识点的局部上下文
3. 对评委和教师而言，“概念/方法/现象”比“词语级实体”更可解释
4. 节点数量可控，适合图谱可视化交互

#### 当前限制

1. 当前粒度主要依赖 LLM 输出质量
2. 不同教材的章节标题风格不一致时，粒度一致性仍会受影响

### 2.2 重复判定标准

#### 需求问题

跨教材整合时，必须判断两个知识点是否表示同一概念。  
如果重复判定标准过松，会把不同概念错误合并；  
如果过严，会把同义知识点保留成多个节点。

#### 当前项目的实际方案

当前实现使用“名称标准化 + 同章映射 + 跨教材合并”三层策略。

#### 1. 单教材内部重复判定

在同一本教材图谱构建阶段，系统对知识点名称做标准化处理：

- 去除空白符
- 转小写

即：

```python
canonical_name = re.sub(r"\s+", "", value).lower()
```

若两个节点标准化后名称一致，则视为重复知识点，合并处理。

合并时保留：

1. 较长的 `definition`
2. 累积出现频次 `frequency`
3. 所属章节集合 `chapters`
4. 原文出处记录 `mentions`

#### 2. 跨教材重复判定

当多本教材都已经生成各自图谱后，系统构建组合图谱时继续使用相同的标准化名称作为合并键。

组合图谱中会保留：

- `source_textbooks`：该知识点出现于哪些教材
- `source_count`：出现于多少本教材
- `frequency`：在所有教材中的累计出现次数

#### 3. 当前判定标准的优点

1. 实现简单，计算稳定
2. 对明显重复概念有效
3. 可以直接支撑前端“频次可视化”和“教材来源区分”

#### 4. 当前判定标准的不足

当前项目尚未实现：

1. 同义词消歧
   - 如“有限冲激响应滤波器”与“FIR滤波器”
2. 上下位概念区分
   - 如“细胞”与“神经细胞”
3. 定义语义相似度判定

因此当前重复判定是“工程上可运行”的第一版，而不是最终最优语义标准。

### 2.3 教学连贯性保障

#### 需求问题

教材整合不是简单拼接文本。  
如果系统只按分数抽取句子，可能破坏学习顺序，导致先讲应用后讲概念，或先讲结论后讲条件。

因此必须回答：系统如何保证整合结果和问答依据具有教学连贯性。

#### 当前项目的实际方案

当前实现采用“结构优先、顺序回排、前置关系显式化”的策略。

#### 1. 正文解析阶段保留教材原始结构

系统在解析时保留：

- `chapter_id`
- `section_id`
- `page_start / page_end`
- 章节/小节正文

这保证后续摘要、图谱和问答都不是脱离结构的纯文本处理。

#### 2. 整合摘要阶段先打分，再按原顺序回排

当前摘要生成流程是：

1. 从各教材小节中切句
2. 计算每句的重要性得分
3. 先选高分句
4. 最终按以下顺序重新排序：
   - 教材顺序
   - 小节顺序
   - 句子顺序

也就是说，摘要选择是“按重要性筛选”，但展示是“按原教学顺序恢复”。

这是一种折中方案：

- 避免纯顺序摘要信息密度不足
- 避免纯打分摘要教学逻辑混乱

#### 3. 图谱阶段通过关系类型补足教学依赖

在知识图谱中，系统显式抽取以下关系类型中的至少三种：

- `prerequisite`
- `parallel`
- `contains`
- `applies_to`

其中 `prerequisite` 直接服务于教学连贯性：  
它把“先学什么、后学什么”的依赖关系显式表达出来。

#### 4. RAG 阶段以小节块为单位检索

问答检索不是按整本书，也不是按单句，而是按小节切块检索。  
这样返回的引用片段通常保留一个相对完整的讲解上下文，减少答复断裂。

#### 5. 当前连贯性策略的边界

当前系统尚未实现：

1. 基于知识图谱的自动教学路径重排
2. 根据 prerequisite 自动生成学习顺序
3. 基于教师反馈的章节级再编排

因此目前连贯性保障主要依赖：

- 结构保留
- 顺序回排
- 图谱中的前置关系

### 2.4 压缩比计算方式

#### 需求问题

项目要求将多本教材压缩到不超过原始体量 `30%` 的精华版本。  
因此需要定义“体量”和“压缩比”的计算方法。

#### 当前项目的实际方案

当前系统采用“字符数预算”的方式计算压缩比。

#### 1. 原始体量定义

原始体量不是按 PDF 文件字节数，而是按参与摘要候选句的总字符数计算：

```text
original_characters = sum(len(sentence_text))
```

即，系统先从所有教材小节中切分句子，再统计所有有效句子的字符数之和。

#### 2. 目标压缩预算

目标预算定义为：

```text
target_budget = original_characters * summary_ratio
```

其中：

- 默认 `summary_ratio = 0.3`
- 当前代码对输入进行约束：`0.1 <= ratio <= 0.5`

#### 3. 摘要实际长度

系统按句子重要性从高到低选句，直到累计字符数达到预算：

```text
summary_characters = sum(len(selected_sentence))
```

返回结果中同时给出：

- `original_characters`
- `summary_characters`
- `summary_ratio`

#### 4. 为什么选择字符数而不是页数

1. 页数受排版影响太大
2. 文件大小受图片、字体等非正文因素影响太大
3. 以字符数衡量更贴近“信息承载量”

#### 5. 当前压缩比策略的特点

优点：

1. 实现简单
2. 可解释性强
3. 便于和前端展示对接

不足：

1. 中文字符数不能完全等价于知识密度
2. 句子级截取可能让一些高价值长段落吃亏

---

### 2.5 RAG 分块策略选择依据

#### 需求问题

RAG 的检索效果很大程度上取决于分块策略。  
块太大，会降低检索精度；  
块太小，会丢失上下文。

因此必须明确：当前项目为什么采用现在的分块方式。

#### 当前项目的实际方案

当前系统使用“按小节结构切块 + 固定长度滑窗”的策略。

具体参数来自当前代码配置：

- `chunk_size = 900`
- `chunk_overlap = 150`

#### 1. 分块流程

1. 先按章节/小节结构提取 `sections`
2. 对每个 `section.text` 单独切块
3. 每块最多约 `900` 字符
4. 相邻块保留 `150` 字符重叠

#### 2. 选择依据

##### 结构边界优先

系统不跨小节直接拼块，而是先在小节内部切块。  
这样做的原因是：

1. 小节本身就是较稳定的教学语义边界
2. 可以减少“块中主题混杂”
3. 有利于回答时返回更可解释的出处

##### 固定长度上限

使用固定上限 `900` 字符，是为了平衡：

1. 检索精度
2. 向量或 TF-IDF 表示稳定性
3. 回答时上下文充分性

##### 重叠区设计

设置 `150` 字符 overlap，是为了减少知识点刚好落在块边界时的断裂问题。

#### 3. 当前检索方式

当前实现使用：

- `TF-IDF`
- `cosine similarity`

并从得分最高的块中选取 `top_k` 作为引用候选。

#### 4. 为什么当前策略适合第一版实现

1. 不依赖外部向量数据库
2. 在本地即可运行
3. 结构简单，容易调试
4. 对教材类长文本已经具有基本可用性

#### 5. 当前局限

1. 还未引入基于知识点图谱的检索增强
2. 还未引入语义向量检索与 rerank
3. 固定块长对不同学科文本未做差异化优化

---

## 3. 需求结论

结合当前项目实现，五个关键子问题已经有可运行的工程答案：

1. **知识点粒度定义**  
   采用“章节级输入 + 核心知识点输出”的粒度控制，保证图谱可解释、可视化可控。

2. **重复判定标准**  
   采用基于标准化名称的节点合并策略，并保留频次、来源教材和出处记录。

3. **教学连贯性保障**  
   通过结构保留、摘要顺序回排和 prerequisite 关系抽取保障基本教学顺序。

4. **压缩比计算方式**  
   采用基于正文有效句字符数的预算控制，默认目标比例为 `30%`。

5. **RAG 分块策略选择依据**  
   采用“小节边界优先 + 固定长度滑窗 + 重叠补偿”的策略，兼顾检索精度与上下文完整性。

这五项方案共同构成了当前项目的“第一版可运行实现”。  
它们已经足以支撑比赛展示、系统演示和评委阅览，同时也明确暴露了下一阶段可继续优化的方向。
