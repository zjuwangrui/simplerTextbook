# 多教材知识整合系统阶段性报告

## 1. 报告说明

本项目原计划以 7 本教材为样本，完成教材解析、知识图谱构建、跨教材对比、整合摘要生成、RAG 问答和分析报告输出，并据此形成完整的整合分析案例。

但从当前项目进展来看，系统尚未完成稳定的“7 本不同教材端到端闭环验证”，因此本报告不再将重点放在“7 本教材整合效果对比”上，而是调整为**阶段性实现报告**。报告主要说明：

1. 当前系统已经完成了哪些功能。
2. 已完成功能的实现方式和可验证结果。
3. 当前尚未完成的部分及其原因。
4. 下一阶段的优化方向。

这种写法能够避免在项目尚未完全完成时对结果做过度推断，同时也更符合当前代码与运行产物的真实状态。

## 2. 项目目标

`SimplerTextbook` 面向“多教材知识整合”场景，目标是构建一个可运行的教学辅助系统，支持以下能力：

1. 上传多本教材并完成正文解析。
2. 对教材章节与小节进行结构化处理。
3. 为单本教材构建知识图谱并进行可视化展示。
4. 对多本教材进行主题重叠、差异和缺失分析。
5. 生成压缩后的整合摘要。
6. 基于教材原文完成 RAG 问答。
7. 输出 Markdown 格式分析报告。

## 3. 当前已完成内容

结合当前仓库代码、测试脚本和 `runtime` 目录中的运行结果，可以确认系统已经完成以下阶段性功能：

### 3.1 教材上传与正文解析

后端已经实现教材上传、文件落盘、逐步解析和状态记录流程，支持 `PDF / Markdown / TXT / Word.docx` 等格式。解析结果会保留教材的章节、小节、文本块和关键词等结构化信息。

当前仓库中已经存在实际解析结果，说明该流程已经可以跑通。例如：

1. `runtime/backend/data/textbooks.json` 中已经记录多条教材解析数据。
2. 其中 1 本 `62` 页教材已完成解析，统计结果包括 `52415` 字符、`51` 个小节、`93` 个文本块。
3. 多条大体量 PDF 教材也已完成解析，说明系统具备处理长文档的基础能力。

### 3.2 单教材知识图谱生成

项目已实现按章节调用 LLM 抽取知识点与关系，并将结果组织为前端可展示的图谱数据。当前仓库中的以下产物能够证明该功能已具备初步可用性：

1. `tests/output/image.json`
2. `src/backend/server/services/graph_service.py`
3. `runtime/backend/data/textbooks.json` 中某教材的 `graph_status = ready`

这说明“教材解析 -> 图谱生成 -> 图谱结果保存”这一链路已经至少在单教材场景下被验证。

### 3.3 跨教材分析与整合摘要

项目中已经实现跨教材分析和整合摘要的基础逻辑：

1. `AnalysisService` 根据教材关键词计算共享主题、差异主题、缺失主题和两两相似度。
2. `IntegrationService` 对多本教材的小节文本切句、打分，并在给定压缩比例下生成整合摘要。
3. `ReportService` 可以把教材概况、跨教材分析、整合摘要和教师对话历史组织成 Markdown 报告。

说明当前系统在代码层面已经具备“多教材整合”的基本框架，只是还缺少完整稳定的样本验证。

### 3.4 RAG 问答与前端集成

项目已实现基于 `TF-IDF + cosine similarity` 的检索式问答逻辑，前端也已经集成教材列表、图谱展示、跨教材分析、摘要与报告入口。说明当前系统并非停留在单一算法脚本层面，而是形成了可演示的前后端原型。

## 4. 当前可验证的阶段性结果

由于无法直接给出“7 本教材整合效果”的完整实验结论，本报告重点采用“可验证中间结果”的方式呈现当前进度。

### 4.1 已有运行产物

当前仓库中可以直接作为证据的文件包括：

1. `tests/output/text.json`：教材正文解析输出。
2. `tests/output/image.json`：知识图谱生成输出。
3. `runtime/backend/data/textbooks.json`：教材元数据、解析状态、图谱状态。
4. `runtime/backend/uploads/`：实际上传教材文件。

这些文件说明项目并非仅完成界面和接口设计，而是已经产生了真实的中间数据。

### 4.2 已暴露的问题

当前运行结果同时也暴露出项目尚未完成的部分：

1. `runtime/backend/data/textbooks.json` 中存在教材解析失败记录，失败原因是外部模型调用超时。
2. 当前样本中存在重复教材记录，尚不能代表“7 本不同教材”的整合场景。
3. `runtime/backend/data/reports.json` 和 `runtime/backend/data/dialogue.json` 仍为空，说明教师反馈与最终报告闭环还没有形成有效样本。

因此，若此时直接撰写“7 本教材整合效果分析”，结论会缺乏充分支撑。

## 5. 为什么本阶段不适合写“7 本教材整合效果分析”

当前不适合硬写完整案例分析，主要有三个原因：

1. **样本不完整**  
   目前没有形成稳定、完整、互不重复的 7 本教材实验集。

2. **流程尚未完全闭环**  
   虽然解析、图谱、摘要、问答和报告模块都已实现基础版本，但多模块联调后的最终结果还不够稳定。

3. **缺少系统性评价指标**  
   当前项目更偏向“先实现可运行原型”，还没有完成针对整合质量、摘要质量、问答准确率的成体系评测。

因此，本阶段更合理的写法是：**把报告定位为系统阶段性实现与初步验证报告，而不是最终效果评估报告。**

## 6. 本阶段报告建议写法

如果需要提交当前版本报告，建议按照以下逻辑展开：

### 6.1 写“做了什么”

重点描述系统目标、模块划分、技术路线和当前完成度，例如：

1. 已完成教材解析流程。
2. 已完成单教材知识图谱构建。
3. 已完成跨教材分析与整合摘要的基础算法。
4. 已完成前后端原型联通。

### 6.2 写“证据在哪里”

不要空泛描述，要引用当前仓库里的实际产物，例如：

1. 测试输出 JSON 已生成。
2. 运行目录中已有教材解析结果。
3. 某教材的图谱状态已为 `ready`。
4. 前端已具备报告与图谱展示入口。

### 6.3 写“还没完成什么”

明确说明目前还不能做完整 7 本教材分析，不是因为方案错误，而是因为：

1. 样本集尚未整理完成。
2. 大文件解析与外部模型调用稳定性仍需优化。
3. 多教材语义合并和最终评测仍需补充。

### 6.4 写“下一步怎么做”

建议把后续工作落到明确任务上：

1. 补齐 7 本不同教材样本并完成统一预处理。
2. 优化大文件异步解析与超时重试机制。
3. 改进跨教材同义概念合并策略。
4. 建立整合摘要质量和 RAG 回答质量的评测指标。
5. 完成教师反馈和自动报告的完整闭环验证。
